## 1. 目标

为单个个人用户提供一套轻量的 A 股与场内 ETF 辅助分析系统：

- 服务器在交易时段持续盯盘，Web 后台只通过 Mac 的 SSH 隧道访问。
- 自选股与持仓合计不超过 20 只，行情约每 30 秒刷新。
- 规则引擎负责技术指标、价格区间、保护位、目标位和仓位计算。
- DeepSeek 只解释资讯、公告、催化与风险，不直接决定交易价格和仓位。
- 重要异动和每日总结通过 PushPlus 推送到个人微信。
- 每日收盘后扫描全市场，候选股由规则筛选、AI 复核，用户确认后才能加入自选。
- 控制数据源和模型费用，并在后台显示用量、估算费用和剩余预算。

系统只提供辅助研究，不自动下单，不连接券商账户。

## 2. 明确不做

首版不包含：

- 自动交易、券商账户连接或交易指令提交。
- 多用户、注册、付费、公开分享或公开荐股。
- 港股、美股、期货、期权和场外基金。
- 毫秒级行情、逐笔成交和盘口交易系统。
- 通用回测平台、复杂选股条件编辑器和自然语言选股。
- Redis、Celery、微服务、PostgreSQL 和独立 SPA 前端。
- 公网 Web、域名、ICP备案、Caddy/Nginx 和公网 HTTPS。

## 4. 总体架构

采用“API 优先的模块化单体”：一个代码仓库、一个 Docker 镜像、两个运行进程。

```text
Mac 浏览器
  │ http://127.0.0.1:8000
  ▼
SSH 加密隧道
  ▼
服务器 127.0.0.1:8000
  │
  ├─ Web 进程：FastAPI、Jinja、HTMX、SSE、/api/v1
  └─ Worker 进程：盯盘、资讯、告警、选股、总结、备份
          │
          ├─ Services：指标、策略、持仓风控、建议、预算、通知
          ├─ Providers：行情、Tushare、博查、DeepSeek、PushPlus
          ├─ Repositories：SQLite 数据访问
          └─ SQLite（WAL）
```

Web 和 Worker 使用同一业务层。页面、API 和任务不得直接调用数据库或第三方接口。

### 4.1 为什么保留两个进程

- Web 重启不应停止盯盘任务。
- 避免多个 Uvicorn Worker 重复注册调度任务。
- 后续可把 Worker 替换成队列消费者，而不重写业务服务。
- 两个进程仍属于同一个模块化单体，不引入微服务通信。

### 4.2 B-ready 边界

未来升级前后端分离时：

- 保留 `/api/v1`、Services、Providers、Repositories 和数据库模型。
- 用 Vue/React 替换 Jinja/HTMX 页面。
- SQLite 可通过 Repository 与迁移脚本替换成 PostgreSQL。
- APScheduler Worker 可替换成 Redis + 队列 Worker。

## 6. 功能模块

### 6.1 总览

- 展示市场状态、数据最新时间和数据源健康状态。
- 自选与持仓按告警严重度和异动程度排序。
- 展示重要告警、相关资讯、候选池数量、任务状态和 API 费用。
- 通过 SSE 推送最新行情、告警和任务状态。
- 持仓标的自动计入盯盘集合；自选与持仓去重后最多 20 只。达到上限时拒绝新增并提示先移除其他自选。

### 6.2 股票详情

- 实时行情、日 K、成交量、MA、MACD、RSI、ATR。
- 当前持仓、成本、盈亏、保护位、目标位和仓位上限。
- 公告与新闻时间线，保留来源、发布时间和原文链接。
- 最新规则建议、AI 解释、证据、置信度、数据质量和策略版本。
- “立即分析”按钮；受预算、冷却和并发限制。

### 6.3 选股器

- 每个交易日收盘后扫描全市场。
- 展示策略、规则分数、排名、入选原因、风险和 AI 复核结果。
- 候选只进入候选池，用户点击确认后才能加入自选。
- 保存每次运行的数据日期、策略版本、数据质量和耗时。

### 6.4 持仓

- 手工录入股票、成本价、数量。
- 配置账户总资金和可用现金。
- 显示单标的仓位、组合集中度、浮动盈亏和保护位风险。
- 不连接券商，不自动同步账户。

### 6.5 告警

- 支持价格突破、涨跌幅、量比、均线交叉、MACD、RSI 和重要资讯规则。
- 支持启停、删除、测试、触发历史、通知结果和冷却状态。
- 同一持续条件只在状态变化时触发；恢复后再次越界才允许重新触发。

### 6.6 设置

- 自选、风险档位、预算、调度和数据保留参数。
- 数据源、DeepSeek、博查和 PushPlus 连接测试。
- 密钥只显示“已配置”，不回显原值。
- 系统健康、备份、费用和任务诊断。

## 7. 数据源设计

所有数据源实现统一 Provider 协议，输出标准化模型，业务层不感知第三方字段。

### 7.1 盘中行情

- 主源：腾讯批量网页行情。
- 备用：AKShare 封装的东方财富行情。
- 每次记录 `provider`、`source_timestamp`、`received_at` 和字段完整度。
- 主源连续失败 3 次进入短路，自动切换备用源。
- 两个源都失败或行情超过 90 秒时标记 `stale`，停止产生新建议和新告警。

这些属于公开网页行情，不作为交易所授权实时行情，不对外转售或公开分发。

### 7.2 全市场日线

- A 股：Tushare `daily(trade_date=...)` 按交易日一次获取全市场。
- ETF：AKShare ETF 日线 Provider。
- 初始化保存最近 250 个交易日日线，之后每日增量更新。
- 未配置 Tushare Token 时，实时自选功能可运行，但全市场选股器保持禁用并显示原因。
- 需要更高积分的基本面数据属于增强能力；缺失时禁用依赖该数据的策略，不伪造或默认为零。

### 7.3 公告与资讯

- 巨潮资讯：公司公告权威源，08:30～18:00 每 5 分钟查询自选和持仓。
- 博查 Web Search API：个股新闻、行业事件、政策和舆情。
- 东财个股资讯：博查失败时备用，并用于交叉验证。
- 标题规范化 + URL + 内容摘要哈希去重。
- 来源分为 `official`、`major_media`、`aggregator`、`unknown`。
- AI 摘要不能替代原文；页面和推送必须保留原始链接。

### 7.4 AI

- Provider：DeepSeek OpenAI 兼容接口。
- 默认模型：`deepseek-v4-flash`，模型名可配置。
- 使用结构化 JSON 输出并做 Schema 校验。
- 超时或解析失败时保留规则版结论，AI 部分标记不可用。
- 不把完整历史 K 线发送给模型，只发送本地计算后的结构化指标和必要证据。

### 7.5 微信通知

- PushPlus 官方消息接口。
- 通知任务独立重试，不影响告警事件和分析落库。
- 推送保存渠道、尝试次数、成功状态、错误码和脱敏诊断。

## 9. 告警与降噪

首版告警类型：

- `price_cross`
- `change_pct`
- `volume_ratio`
- `ma_cross`
- `macd_cross`
- `rsi_threshold`
- `important_news`
- `position_stop_proximity`

默认规则：

- 普通价格和技术告警按 `rule + symbol` 冷却 60 分钟。
- 接近保护位告警按状态变化触发，并至少冷却 15 分钟。
- 资讯按 URL/内容哈希永久去重；同一事件的多来源合并展示。
- 告警先落库，再异步调用 AI 和 PushPlus。
- AI 或 PushPlus 失败不能丢失原始规则事件。

## 10. 建议与风险引擎

### 10.1 输入

- 最新有效行情与最近 250 日日线。
- MA20/MA60、EMA、MACD、RSI14、ATR14、量比、支撑与阻力。
- 持仓成本、数量、账户总资金和可用现金。
- 最近公告、新闻、来源等级和事件时间。
- 风险档位和策略版本。

### 10.2 数字计算

本地规则计算：

- 参考关注区间：结构支撑、MA20 回踩区或突破确认区。
- 保护位：最近结构低点与 ATR 缓冲共同确定。
- 第一/第二目标：前高阻力与 1.5R/2R 风险收益位综合确定。
- `risk_amount = account_equity × 1%`。
- `per_unit_risk = entry_mid - stop_price`。
- `qty_by_risk = floor(risk_amount / per_unit_risk / lot_size) × lot_size`。
- `qty_by_cap = floor(account_equity × 20% / entry_mid / lot_size) × lot_size`。
- 最终数量为风险、仓位、可用现金三类上限的最小值。

任何分母无效、行情过期、停牌或数据不足时，不给出数量建议。

### 10.3 AI 权限

DeepSeek 可以：

- 总结公告、新闻、催化与风险。
- 标注来源冲突、消息可信度和数据缺口。
- 解释本地技术信号。
- 因新增风险降低建议等级。

DeepSeek 不可以：

- 修改本地计算出的价格、保护位、目标位和仓位。
- 在没有证据时作确定性预测。
- 将本地“观察/等待”升级成更积极等级。

### 10.4 状态集合

- 回避
- 观察
- 等待回踩
- 允许试仓
- 持有
- 减仓
- 退出

每条建议展示数据时间、来源、计算依据、证据、置信度、数据质量、策略版本和免责声明。

## 11. 选股器

### 11.1 全局过滤

默认排除：

- ST、*ST、退市整理、停牌。
- 上市不足 120 个交易日。
- 股票 20 日平均成交额低于 1 亿元。
- ETF 20 日平均成交额低于 5000 万元。
- 日线不足 120 条或关键字段缺失。

阈值可在设置页调整，但修改后产生新的策略版本。

### 11.2 内置策略

1. 趋势回踩：MA20 > MA60，收盘位于 MA20 附近，RSI14 为 45～60，量能未明显衰减。
2. 放量突破：突破 60 日区间高点，成交量高于 20 日均量 1.5 倍，收盘不过度偏离突破位。
3. 质量趋势：趋势条件成立，并满足可用基本面中的盈利、现金流和营收增长门槛；基本面数据不可用时该策略禁用。
4. ETF 动量：ETF 的 20/60 日趋势、相对强度、回撤和流动性综合排序。

规则先在本地完成过滤和打分。仅综合排名前 10 名调用博查与 DeepSeek，最终进入候选池，不自动加入自选。

## 12. 预算控制

### 12.1 博查

- 每日目标预算：¥5。
- 每日硬上限：¥10。
- 优先级：持仓和重大异动 40%，候选前 10 名 30%，普通自选 20%，手动与重试 10%。
- ¥5～8 降低普通自选频率。
- ¥8～10 只保留持仓、重大异动和手动请求。
- 达到 ¥10 后停止自动调用，继续使用巨潮、东财和缓存。
- 初始设置必须填写当前套餐的单次调用估算单价；未填写时禁用博查自动任务，只允许连接测试。后续按控制台账单校准单价。

### 12.2 DeepSeek

- 月度目标预算：¥10。
- 月度硬上限：¥20。
- 默认只在重大事件、手动分析、收盘候选复核和每日总结时调用。
- 达到硬上限后返回规则版分析；管理员可修改下月预算，但系统不自动突破。
- 费用按接口返回的输入/输出 Token 和后台配置的模型单价计算；模型或官方价格变化时必须先更新单价配置。

### 12.3 计量

- 保存请求用途、Provider、模型、输入/输出 Token、调用次数和估算费用。
- Provider 未返回金额时，按设置中的单价估算并明确标注为估算值。
- Web 展示日/月趋势、命中率和剩余预算。

## 15. 安全设计

### 15.1 网络

- 阿里云中国大陆服务器保留公网 IPv4，仅用于 SSH。
- 安全组只开放 TCP 22，来源固定为用户当前公网 IP 的 `/32`；家庭公网 IP 变化时先在阿里云控制台更新规则。
- 不开放 80、443、8000 或数据库端口。
- Docker 仅发布 `127.0.0.1:8000:8000`。
- Mac 使用 `ssh -L 8000:127.0.0.1:8000 user@server` 访问后台。

### 15.2 SSH 与应用

- SSH 只允许密钥认证，禁用密码登录和 root 直接登录。
- Web 保留单管理员密码作为第二层保护。
- 密码使用 Argon2id 哈希。
- Session Cookie 使用 HttpOnly、SameSite=Strict；写请求使用 CSRF Token。
- 登录失败限速；无操作 30 分钟或最长 8 小时后退出。

### 15.3 密钥

- `APP_ENCRYPTION_KEY` 仅保存在服务器 `.env`，权限 `0600`。
- Web 输入的 DeepSeek、博查、Tushare 和 PushPlus 密钥加密后入库。
- 环境变量可以覆盖加密存储值，便于恢复。
- API 响应、日志、错误和诊断不得回显密钥、完整 Token 或完整请求头。

## 16. 故障处理

| 故障 | 行为 |
| --- | --- |
| 主行情源失败 | 连续 3 次后短路，切换备用源 |
| 主备行情均失败 | 标记 stale，保留最后值但停止新建议与告警 |
| 日线任务失败 | 选股任务失败并可补跑，不影响盘中盯盘 |
| 资讯失败 | 返回技术版建议并标明资讯缺失 |
| AI 超时/格式错误 | 保留规则结论，AI 部分失败，有限重试 |
| PushPlus 失败 | 事件已落库，通知有限重试并保存结果 |
| SQLite 写入失败 | 当前任务失败，不发送未持久化的告警 |
| 服务重启 | 依靠 job key、事件 fingerprint 和通知记录避免重复 |

所有外部请求设置连接/读取超时、有限指数退避和可观察错误码。不得使用无限重试。

## 17. 验收标准

- 交易时段行情约每 30 秒刷新，超过 90 秒明确显示过期。
- 主行情源故障能自动切换；持续条件不会重复刷微信。
- 重大事件从本地触发到 PushPlus 成功不超过 2 分钟。
- 全市场选股在 2 vCPU/4 GiB 上 20 分钟内完成，峰值内存低于 3 GiB。
- 博查每日达到 ¥10、DeepSeek每月达到 ¥20 后硬停止自动调用。
- 服务重启不会重复执行同一交易日选股、AI 复核或每日推送。
- 数据源、AI 和通知单独失败时，其他主链路可继续运行。
- 备份通过真实恢复测试，恢复后自选、持仓、规则、历史和预算记录完整。
- 公网扫描无法访问 8000、80、443；只有 SSH 隧道可以打开后台。

## 18. 测试策略

1. 单元测试：指标公式、风险数量、策略过滤、去重、冷却、预算路由。
2. Provider 契约测试：使用录制且脱敏的响应 Fixture，不依赖实时网络。
3. 数据库/API 集成测试：SQLite 临时库、Alembic、CRUD、事务和鉴权。
4. 调度测试：交易时段、午休、节假日、重启幂等和补跑。
5. Web 关键路径：登录、添加自选、录入持仓、创建规则、查看候选。
6. 在线冒烟：腾讯、东财、Tushare、巨潮、博查、DeepSeek、PushPlus逐个测试。
7. 资源测试：模拟约 6000 个标的、250 日日线，在 4 GiB 环境运行选股。
8. 部署验收：Docker 重启、SSH 隧道、端口暴露检查、备份恢复和日志脱敏。

## 19. 分阶段交付

### Phase 1：基础与实时看板

### Phase 2：规则告警与微信

技术指标、告警规则、冷却/去重、事件历史、PushPlus 和数据源健康。

### Phase 3：资讯与 AI

巨潮、博查、东财资讯、预算路由、DeepSeek结构化解释和股票详情。

### Phase 5：收盘选股

### Phase 6：加固与交付

资源压测、故障演练、备份恢复、费用校准、运维文档和服务器正式部署。

每个 Phase 独立验收，前一阶段通过后再进入下一阶段。

## 20. 风险与回滚

- 免费网页行情可能变更：通过 Provider 隔离、备用源和 stale 保护降低影响。
- 数据源访问可能限流：控制频率、短路和缓存，不使用激进并发。
- AI 可能产生幻觉：数字本地计算、结构化输出、证据链接和只允许降级。
- 2 GiB 配置可能内存不足：最终选择 4 GiB；选股仍采用分批计算。
- 服务器部署失败：本地 Docker 可运行同一镜像，数据库可从备份恢复。
- 单阶段发布出现问题：回滚到上一镜像版本并保留兼容数据库迁移；不可逆迁移必须单独审批。
